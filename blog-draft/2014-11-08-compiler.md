# 编译器的工作过程

源代码要运行，必须先转成二进制的机器码。这就是编译器的任务。

下面是一段C语言代码。

```c

#include <stdio.h>

int main(void)
{
    fputs("Hello, world!\n", stdout);
    return 0;
}

```

编译器先把这段代码转为机器码，然后才能在计算机上运行。这个过程并不简单，涉及许多中间步骤。大多数资料都语焉不详，只说编译源码需要三步。

```bash

$ ./configure
$ make
$ make install

```

本文将介绍编译器的工作过程，也就是上面这三个命令到底在干什么。我主要参考了Alex Smith的文章[《Building C Projects》](http://nethack4.org/blog/building-c.html)。

需要声明的是，本文主要针对gcc编译器，也就是C语言的编译，不一定适用于其他语言的编译。

## 第一步 配置（configure）

软件可能会安装在许多计算机上，它们的系统往往都不尽相同。编译器在开始工作之前，需要知道当前计算机的系统环境，以及用户的安装要求，比如C语言的标准库在哪里、软件的安装位置、需要安装哪些组件等等。这个步骤就叫做“配置”（configure）。

这些信息存放在配置文件中，该文件一般是一个叫做configure的脚本文件，通常它是由[autoconf](http://zh.wikipedia.org/wiki/Autoconf)工具生成的。编译器根据这个文件，进行接下来的编译。

configure脚本已经尽量考虑到不同系统的差异，并且对各种环境变量给出默认值。如果用户的系统环境比较特别，或者有不一样的需求，就需要手动调整configure文件。

```bash

$ ./configure --prefix=/www --with-mysql

```

上面是php源码安装时的一个配置实例，用户指定软件安装在www目录，并且编译时需要mysql模块。

## 第二步 确定标准库和头文件的位置

源代码会用到标准库函数（standard library），以及包含各种头文件（header）。标准库和头文件可以存放在任意目录中，编译器实际上没有办法自动检测它们的位置，只有通过配置文件才能知道。

所以，有了配置文件以后，编译的第二步，就是从中获知标准库和头文件的位置。

一般来说，配置文件会给出一个清单，列出几个具体的目录。等到编译时，编译器就按顺序到这几个目录中，寻找头文件。

## 第三步 确定依赖关系

源代码文件之间，往往存在依赖关系。编译器在这一步，需要确定编译的先后顺序。

这是为了达到以下两个目的。

> - 假定a文件依赖于b文件，确保当b文件发生变化时，a文件会被重新编译。
> 
> - 确保只有在b文件编译完成后，才开始编译a文件。

运行configure脚本之后，会生成一个makefile文件。编译顺序就由这个文件指定，里面列出了哪个文件先编译，哪个文件后编译。

## 第四步 头文件的预编译（precompilation）

头文件往往包含了常用函数的声明。一个头文件，可能会被多个源码文件引用。

为了节省时间，编译器会预编译头文件。需要的时候，就把编译好的头文件，直接插入引用它的源码文件即可。这样保证了头文件只需编译一次，而不用每次用到的时候，都编译一遍。

当然，不是头文件的所有内容，都被预编译。其中，#define命令声明宏的部分，就不会被预编译，因为宏需要以定义时的形式，原封不动地插入代码文件。 

 ## 第五步 预处理（Preprocessing）

编译器在这一步，替换掉源码的头文件和宏。

本文开始时给出的那段源码，引用了头文件stdio.h，替换后的样子如下。

```c

extern int fputs(const char *, FILE *);
extern FILE *stdout;

int main(void)
{
    fputs("Hello, world!\n", stdout);
    return 0;
}

```

头文件往往很长，上面代码只截取了其中与源码相关的那部分，即fputs和FILE的声明。另外，为了便于阅读，上面代码的头文件没有经过预编译，而实际上，插入源码的是预编译后的结果。

这一步完成后的代码，将作为机器码编译的直接依据，所以这一步被称为“预处理”（Preprocessing）。另外，编译器在一步还会移除源码中的注释。

## 第六步 编译（Compilation）

编译器在这一步，将预处理的结果转为机器码。对于某些编译器来说，还存在一个中间步骤，会先把源码转成汇编码（assembly），然后再把汇编码转为机器码。

下面就是示例代码转成的汇编码。

```c
	.file	"test.c"
	.section	.rodata
.LC0:
	.string	"Hello, world!\n"
	.text
	.globl	main
	.type	main, @function
main:
.LFB0:
	.cfi_startproc
	pushq	%rbp
	.cfi_def_cfa_offset 16
	.cfi_offset 6, -16
	movq	%rsp, %rbp
	.cfi_def_cfa_register 6
	movq	stdout(%rip), %rax
	movq	%rax, %rcx
	movl	$14, %edx
	movl	$1, %esi
	movl	$.LC0, %edi
	call	fwrite
	movl	$0, %eax
	popq	%rbp
	.cfi_def_cfa 7, 8
	ret
	.cfi_endproc
.LFE0:
	.size	main, .-main
	.ident	"GCC: (Debian 4.9.1-19) 4.9.1"
	.section	.note.GNU-stack,"",@progbits
```

转好的代码称为对象文件（object file）。

## 第七步 连接（Linking）

下一步就是要将对象文章，包装成可执行文件。

如果你仔细看上一步的转码结果，会发现其中引用了stdout函数和fwrite函数。也就是说，程序要正常运行，除了上面的代码以外，还必须有stdout和fwrite这两个函数的代码，它们是由C语言的标准库提供的。所以，必须把这两个函数的代码也包括在可执行文件之内。

编译器的下一步工作，就是先统计用到了哪些外部函数，然后把它们（通常是后缀名为.lib和.a的文件）添加进来。这就叫做连接（linking）。这种将外部函数拷贝添加的方式，叫做[静态连接](http://zh.wikipedia.org/zh-cn/%E9%9D%9C%E6%85%8B%E9%80%A3%E7%B5%90%E5%87%BD%E5%BC%8F%E5%BA%AB)（static linking），后文会提到还有[动态连接](http://zh.wikipedia.org/wiki/%E5%8A%A8%E6%80%81%E9%93%BE%E6%8E%A5%E5%BA%93)（dynamic linking）。

make程序就用来完成这一步。

## 第八步 安装（Installation）

上一步的连接是发生在内存当中，即编译器在内存中生成了可执行文件。我们必须以某种形式，将可执行文件保存下来。

表面上，这一步比较简单，就是将可执行文件保存到硬盘上用户事先指定的安装目录，连带与其相关的数据文件。但是，实际过程中，这一步还必须完成创建目录、保存文件、设置权限等步骤，整个过程就称为“安装”（Installation）。

## 第九步 操作系统连接

可执行文件安装后，必须以某种方式通知操作系统，让其知道可以使用这个程序了。比如，我们安装了一个文本阅读程序以后，往往希望双击txt文件，该程序就会自动运行。

所以，我们必须在操作系统中登记这个程序，也就是登记一些元数据：文件名、文件描述、关联后缀名等等。在Linux操作系统中，这些信息通常保存在/usr/share/applications目录下的一些 .desktop文件中。另外，在Windows操作系统中，还需要在Start启动菜单中，建立一个快捷方式。

整个这一步就叫做“操作系统连接”。make install命令，就用来完成上一步和这一步。

## 第十步 生成安装包

只有很少一部分用户，愿意耐着性子，从头到尾编译源码。事实上，如果你只有源码可以交给用户，他们会认定你是一个不友好的家伙。大部分用户要的是一个可执行程序，立刻就能运行。这就要求开发者，将上一步生成的可执行文件，做成可以分发的安装包。

所以，编译器还必须有生成安装包的功能。通常是将可执行文件（连带相关的数据文件），以某种目录结构，保存成压缩文件包，交给用户。

## 第十一步 动态连接（Dynamic linking）

正常意义上，编译器的工作到上一步，就已经结束了。程序运行过程中发生的事情，与编译器无关。

但是，在编译过程中，开发者可以选择，可执行文件包含外部函数库的方式，到底是静态连接还是动态连接。为了文章的完整，这把这一点也包括进来。

前面已经说过，静态连接就是把外部函数库，拷贝到可执行文件中。这样做的好处是，适用范围比较广，不用担心用户机器缺少某个库文件；缺点是安装包会比较大，而且多个应用程序之间，无法共享库文件。

动态连接的做法正好相反，外部函数库不进入安装包，只在运行时动态引用。好处是安装包会比较小，多个应用程序可以共享库文件；缺点是用户必须事先安装好库文件，而且版本和安装位置都必须符合要求，否则就不能正常运行。

现实中，大部分软件采用动态连接，共享库文件。这种动态共享的库文件，Linux平台是后缀名为.so的文件，Windows平台是.dll文件，Mac平台是.dylib文件。
